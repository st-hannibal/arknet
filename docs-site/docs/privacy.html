Privacy & Security

arknet is designed so your prompts stay private at every layer. The network is pure peer-to-peer — there are no HTTP endpoints, no gateways, no middlemen. Three tiers of protection, from transport encryption to hardware-enforced confidentiality.

Pure P2P architecture

There is no HTTP anywhere in the stack. The SDK joins the libp2p gossip mesh directly via validator seed nodes, discovers compute nodes through PoolOffer gossip messages, and sends inference requests through the mesh using libp2p relay and hole punching. Every connection is Noise-encrypted end-to-end.

Two node types only. Validators (consensus + seed + relay) and compute nodes (inference + gossip). There are no routers or gateways. This means fewer parties touch your data and fewer attack surfaces exist.

Three-tier privacy model

Tier Protection Available at genesis What it protects
Transport encryption Noise protocol (libp2p) Yes All P2P traffic encrypted end-to-end. Network eavesdroppers see nothing.
Prompt isolation Protocol design Yes Only the assigned compute node sees your prompt. Validators relay opaque encrypted bytes they cannot read. Verifiers see hashes and token counts — never content. Consensus never touches prompt data.
Confidential inference (TEE) Hardware enclaves (Intel TDX / AMD SEV-SNP) Yes (protocol ready, hardware-dependent) Even the compute node's host OS cannot see your prompt. The model runs inside a Trusted Execution Environment — encrypted memory that only the CPU can access.

How data flows through the network

Every inference request follows the same path. At no point does any party other than the compute node see your prompt in plaintext.

SDK (your app)                   Validator (seed/relay)         Compute Node
  |                                    |                              |
  | 1. Join gossip mesh                |                              |
  |    via seed node                   |                              |
  |----------------------------------->|                              |
  |                                    |                              |
  | 2. Discover compute via            |                              |
  |    PoolOffer gossip                |                              |
  |<---------------------------------->|                              |
  |                                    |                              |
  | 3. Send inference request          |                              |
  |    (Noise-encrypted e2e)           |                              |
  |----------------------------------->| relay (opaque bytes)         |
  |                                    |----------------------------->|
  |                                    |                              |
  |                                    |         4. Decrypt, run      |
  |                                    |            inference         |
  |                                    |                              |
  | 5. Tokens stream back             |                              |
  |    (Noise-encrypted e2e)           |                              |
  |<-----------------------------------|<-----------------------------|
  |                                    |                              |
  |                                    | 6. Receipt submitted         |
  |                                    |    to chain for settlement   |

The validator in the middle acts as a relay — it forwards encrypted bytes between the SDK and the compute node via libp2p relay and hole punching. It cannot decrypt the payload. It sees only metadata needed for routing: peer IDs and connection info.

Session keys: limiting exposure

Your wallet's master key (~/.arknet/wallet.key, Ed25519) never touches the network directly. Instead, you create ephemeral session keys with a spending limit and expiry:

from arknet_sdk import Wallet, Client

wallet = Wallet.create()
wallet.save()  # saved to ~/.arknet/wallet.key

session = wallet.create_session(
    spending_limit=100_000_000,  # 0.1 ARK max
    expires_secs=3600            # 1 hour
)
client = Client.connect(session=session)

If a session key is compromised, the attacker can spend at most the session's allowance. Your master wallet is unaffected. Session keys are the recommended way to interact with the network for daily use.

How prompt encryption works (TEE)

When a user sends a request to a TEE-capable compute node, the prompt never exists in plaintext outside the hardware enclave.

User                          Enclave (inside TEE)         Host OS
  |                                |                          |
  | 1. Get enclave public key      |                          |
  |   (from chain state)           |                          |
  |<----------------------------->|                          |
  |                                |                          |
  | 2. Encrypt prompt to           |                          |
  |    enclave pubkey              |                          |
  |    (X25519 + ChaCha20)         |                          |
  |------------------------------->| 3. Decrypt inside        |
  |                                |    enclave               | sees: encrypted blob
  |                                | 4. Run inference         | sees: nothing
  |                                |    (in enclave memory)   |
  | 5. Tokens stream back          |                          |
  |    over Noise-encrypted P2P    |                          |
  |<-------------------------------|                          | sees: encrypted blob

The full flow:

  1. User generates an ephemeral X25519 keypair — one per request, discarded after.
  2. ECDH with the enclave's public key produces a shared secret. The enclave's pubkey is published on-chain when the node registers.
  3. HKDF-SHA256 derives a symmetric key from the shared secret.
  4. ChaCha20-Poly1305 encrypts the prompt — authenticated encryption, tamper-evident.
  5. The enclave decrypts using its private key, which was generated inside the TEE and never leaves enclave memory.
  6. Inference runs entirely in enclave memory. The host OS, hypervisor, and other processes cannot read it.
  7. Response tokens stream back over the Noise-encrypted P2P channel.

Cryptography: All primitives are from audited Rust crates — x25519-dalek, chacha20poly1305, hkdf, sha2. No custom cryptography.

Using confidential inference

Pass prefer_tee=True and the network will route your request only to TEE-capable compute nodes.

from arknet_sdk import Wallet, Client

wallet = Wallet.create()
session = wallet.create_session(spending_limit=100_000_000, expires_secs=3600)
client = Client.connect(session=session)

response = client.infer(
    model="Qwen/Qwen3-0.6B-Q8_0",
    prompt="Confidential question here",
    max_tokens=64,
    prefer_tee=True,
)

No silent downgrades. When prefer_tee=True, the network only picks TEE-capable nodes. If none are available, you get a clear error — the request is never silently sent to a non-TEE node.

For node operators: enabling TEE

If your hardware supports Intel TDX or AMD SEV-SNP, you can register as a TEE-capable compute node in five steps.

  1. Generate enclave keypair: 
    arknet tee keygen
  2. Get attestation quote from your TEE hardware. This proves your CPU supports a genuine enclave.
  3. Register on-chain: 
    arknet tee register --platform intel-tdx --quote-file quote.bin
  4. Enable in config (~/.arknet/node.toml): 
    [tee]
enabled = true
platform = "intel-tdx"
  5. Restart your node. It is now advertised as TEE-capable and receives confidential inference requests via the gossip mesh.

What TEE does NOT protect against

Honesty about limitations:

  • Metadata is visible. TEE does not hide which model you are querying or how many tokens were generated. These values appear in inference receipts — they are required for verification and reward calculation.
  • CPU side-channel attacks. TEE does not protect against side-channel vulnerabilities in specific CPU implementations (e.g., speculative execution attacks). Intel and AMD patch these as they are discovered, but they exist in the threat model.
  • Genesis-era attestation is structural. At genesis, TEE quote verification checks the quote structure and signature format. Full root-of-trust verification against Intel/AMD certificate authorities will be activated once the verification library is deployed and audited. This is a known gap documented in the security model.

Wallet security

Your wallet key is Ed25519, stored at ~/.arknet/wallet.key. Your on-chain address is the first 20 bytes of blake3(pubkey).

  • Never share your wallet key. Anyone with the file can spend your ARK.
  • Use session keys for daily operations. They limit blast radius if compromised.
  • Back up your wallet key. There is no recovery mechanism — if you lose it, those tokens are gone.
